# Методика релевантности и приоритета QAZ.FUND

Дата: 2026-07-16  
Модель: `qazfund-relevance-v2`

## Назначение

Модель помогает упорядочить публичные возможности для Казахстана и Центральной
Азии. Она отвечает на два разных вопроса:

1. `relevance` - насколько запись уместна в каталоге по теме и географии;
2. `priority` - насколько уместную запись удобно брать в работу сейчас с учетом
   оставшегося срока.

Оба значения находятся в диапазоне от `0` до `1`. Это не вероятность получения
финансирования, не проверка юридического соответствия и не персональная рекомендация.
До появления размеченной выборки балл нельзя называть точностью или вероятностью.

## Почему модель изменена

Предыдущая формула независимо складывала каждое совпавшее ключевое слово. Синонимы
одной темы могли учитываться несколько раз, широкий текст быстро достигал `1.0`, а
срок подачи смешивался с содержательной релевантностью. Кроме того, API сохранял
максимум между старым сохраненным баллом и новым расчетом, поэтому после изменения
формулы устаревший высокий балл мог остаться в выдаче.

Версия v2 устраняет эти свойства:

- синонимы объединены в тематические группы и учитываются один раз;
- заголовок и редакционные теги имеют полный вес, совпадение только в описании -
  коэффициент `0.55`;
- тематическая часть ограничена значением `0.56`;
- география и подтвержденный локальный источник считаются отдельно;
- жесткое исключение всегда дает нулевой результат;
- глобальный агрегатор без доказанной применимости не проходит публичный порог;
- срок влияет только на оперативный приоритет;
- API пересчитывает балл текущей версией и не наследует максимум старой модели.

## Формула релевантности

```text
thematic = min(0.56, sum(one_weight_per_matched_concept))
relevance = min(1.0, thematic + geographic + source_context)
```

Тематические группы и их веса:

| --- | ---: |
| ИИ | 0.28 |
| Сельское хозяйство и продовольствие | 0.24 |
| Цифровая инфраструктура | 0.22 |
| Здоровье животных | 0.22 |
| Климат и окружающая среда | 0.22 |
| Медиа и гражданское общество | 0.22 |
| Поддержка бизнеса | 0.20 |
| Государственное управление | 0.20 |
| Образование и навыки | 0.18 |
| Стартапы и инновации | 0.18 |
| Финансирование и рост | 0.14 |

Географические веса:

| Сигнал | Вес |
| --- | ---: |
| Казахстан | 0.30 |
| Центральная Азия | 0.24 |
| Явно глобальная возможность | 0.10 |
| Другой региональный сигнал | 0.08 |
| География не указана | 0.00 |

Проверенный локальный адаптер добавляет `0.06`. Это отражает подтвержденный контекст
источника, но не заменяет проверку критериев конкретной программы.

## Ограничения доверия

- Совпадение с жестким географическим или продуктовым исключением дает
  `relevance = 0` и `priority = 0`.
- Записи глобальных мостовых источников без сигнала Казахстана или Центральной Азии
  получают состояние `review_required` и ограничиваются значением `0.29`.
- Публичный порог каталога равен `0.30`.
- Причина исключения, версия модели, компоненты и совпавшие темы возвращаются в
  `raw.ranking` для проверки оператором и машинным потребителем.

## Формула оперативного приоритета

```text
priority = min(1.0, relevance * 0.90 + deadline_actionability)
```

| Оставшийся срок | Добавка |
| --- | ---: |
| Уже закрыто | 0.00 |
| 0-2 дня | 0.02 |
| 3-7 дней | 0.05 |
| 8-21 день | 0.10 |
| 22-45 дней | 0.08 |
| 46-90 дней | 0.05 |
| Более 90 дней | 0.02 |
| Бессрочно | 0.04 |
| Срок неизвестен | 0.00 |

Окно `8-21 день` получает наибольшую добавку: возможность уже требует действия, но
еще оставляет реалистичный срок на проверку условий и подготовку. Запись с
`review_required` не может подняться выше своей релевантности только из-за дедлайна.

## Похожие возможности

Похожие карточки ранжируются отдельной нормированной функцией:

- пересечение тематических и структурных признаков по Jaccard - `40%`;
- тот же фонд - `30%`;
- тот же формат - `15%`;
- тот же источник - `10%`;
- релевантность кандидата - `5%`.

Минимальный порог сходства - `0.20`. Первый проход ограничивает выдачу одной карточкой
на источник, затем при необходимости заполняет свободные места. Это снижает эффект,
при котором один крупный агрегатор вытесняет остальные источники.

## Сопоставление с практиками и исследованиями

- Instrumentl связывает совпадение с конкретным проектом, а затем переводит выбранные
  возможности в рабочий трекер. QAZ.FUND пока не имеет подтвержденного профиля проекта,
  поэтому не имитирует персональную вероятность совпадения.
- GrantForward разделяет поиск, профиль пользователя, рекомендации, списки и историю.
  Для QAZ.FUND это аргумент в пользу отдельного будущего персонального слоя, а не
  скрытого смешивания профиля с публичным каталогом.
- Исследование 2026 года по сопоставлению исследователей и конкурсов использует
  несколько представлений профиля, нормализацию внутри пользователя и процентильное
  ранжирование. Эти методы уместны после появления реального профиля организации.
- Исследования рекомендательных систем показывают необходимость сохранять баланс
  релевантности и разнообразия. Поэтому похожие карточки диверсифицируются по источнику.
- Работы по калибровке ранжирования подтверждают, что шкалу нельзя трактовать как
  вероятность без тестового набора и калибровки. В интерфейсе используется термин
  «релевантность каталога».
- NIST рекомендует связывать метрики с конкретным контекстом использования, проверять
  их на реалистичном тестовом наборе и документировать ограничения.

## Доказательный этап

Инструментарий подготовки и проверки размеченного набора реализован в
`scripts/ranking_evaluation.py`, а порядок работы описан в
[`RANKING_EVALUATION.md`](RANKING_EVALUATION.md). Перед переходом к обучаемой или
персональной модели все еще нужен согласованный набор не менее 200 реальных записей.
Два независимых редактора должны оценить применимость и достаточность срока, а
расхождения - пройти согласование.

Минимальный отчет должен включать:

- `Precision@10`, `Recall@50` и `nDCG@10` отдельно для релевантности и приоритета;
- покрытие источников и долю уникальных источников в первой десятке;
- ошибки отдельно по аудиториям, форматам, регионам и источникам;
- стабильность результата после обновления источников;
- долю ручных отмен исключения и состояния `review_required`;
- только после накопления исходов - отдельную калибровку, не смешанную с релевантностью.

До выполнения этого этапа изменение весов требует теста, измерения распределения на
полном каталоге и записи новой версии модели. Откат v2 выполняется обычным `git revert`;
миграция базы не требуется, поскольку API пересчитывает текущий балл при чтении.
