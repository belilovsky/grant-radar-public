# QAZ.FUND: план закрытия хвостов

Дата ревизии: 2026-07-15.

## Цель

Этот документ отделяет подтверждённые технические хвосты от продуктовых идей,
которые нельзя честно включить без дополнительных контрактов. Источник истины
для кода — `grant-radar-public`; состояние интеграций подтверждается публичными
`/.well-known/`-контрактами и выпускными проверками, а не планом или статусом в
документации.

## Закрывается в текущем выпуске

1. **Свежесть пустых источников.** Каждый цикл парсера записывает отдельный
   успешный или ошибочный запуск в существующую таблицу `runs`. Поэтому источник,
   который успешно проверен, но не вернул записей, больше не выглядит как
   непроверенный. Публичный `/status` показывает время последней успешной
   проверки, не раскрывая операторские ошибки.
2. **Жизненный цикл HTTP-клиентов.** Планировщик закрывает только созданные им
   парсеры и продолжает закрытие остальных клиентов, если один `close()` завершился
   ошибкой. Переданные извне парсеры остаются под управлением вызывающего кода.
3. **Управляемый dashboard-модуль.** Локализованные тексты и статические стили
   вынесены из `api/dashboard.py` в отдельные модули. Рендер RU и EN проверен
   побайтно: длина и SHA-256 HTML совпадают с состоянием до разделения.
4. **Регрессионные тесты.** Добавлены проверки успешного пустого цикла, ошибки
   источника, закрытия клиентов, выбора свежести по последней проверке и границ
   dashboard-модулей.
5. **Воспроизводимые зависимости.** Транзитивные runtime-зависимости vendored
   QazStack wheel закреплены проверенными версиями; чистый `pip install`,
   `pip check` и `pip-audit` входят в gate.
6. **Runbook и release evidence.** Операционная документация объясняет, что
   свежесть пустого источника появляется после реального успешного цикла; ручная
   подстановка времени не допускается.

## Уже реализовано и не требует повторной разработки

- каталог, фильтры, глубокие ссылки, локальные сохранения и этапы работы;
- CSV, календарь и переносимая резервная копия локального рабочего пространства;
- карточки возможностей, профили фондов, похожие записи и рабочая справка;
- RU/EN, адаптивный интерфейс и AV DS 4 SSR-адаптер;
- источник, доказательный статус, `/coverage`, `/status` и операторская проверка;
- JSON, cache-aware NDJSON, digest, OpenAPI, `llms.txt` и `site-discovery.json`;
- QazStack 1.37.2 для нейтральной нормализации, evidence, diversity и экспорта;
- QazPipe/QazLake read-only boundary, опубликованный как interface-ready, без
  ложного заявления о включённой записи в общий контур.

## Готово в коде, но заблокировано выпускным контуром

Интеграция QazCompute находится в отдельном PR QAZ.FUND `#6`. Она не входит в
этот независимый выпуск. Merge и production deploy разрешены только после
зелёного CI QazCompute PR `#30`, проверки актуального `main`, локальной parity
проверки и зелёного CI самого QAZ.FUND. Ошибка billing/runner не обходится, а
статус `candidate-not-enabled` в ecosystem contract не меняется вручную.

## Условный продуктовый backlog

Эти функции полезны, но не являются «забытым кодом» и не должны появляться как
демонстрационные кнопки:

| Возможность | Обязательное условие до реализации |
| --- | --- |
| Подписки и уведомления | аккаунт, явное согласие, канал доставки, журнал, отключение и удаление данных |
| История изменений источника | версионированные снимки, время наблюдения и воспроизводимый diff |
| Сравнение возможностей | минимальное покрытие сопоставимых полей и явное представление пропусков |
| Командная работа | ACL, владельцы, хранение, экспорт и удаление данных |
| Юридический eligibility score | утверждённая методика, версия норм и журнал обоснований |
| AI-рекомендации | объяснимое правило, ссылка на источник, оценка риска и набор eval-сценариев |
| Карта QazGeo | проверенные координаты или авторитетный идентификатор геометрии региона |
| Запись в QazLake | одобренная схема, retention, idempotency и dry-run QazPipe-коннектора |

## Не делать

- не подставлять фиктивное время проверки пустому источнику;
- не считать ссылку на источник юридической верификацией;
- не переносить продуктовую политику релевантности и редакционные решения в
  QazStack;
- не менять публичную форму API ради унификации пагинации;
- не публиковать карту на основании географии, извлечённой из свободного текста;
- не включать QazCompute, QazPipe или QazLake только записью в документации.

## Выпускные критерии

1. `make lint` и полный `pytest` зелёные.
2. Checksum QazStack wheel, `compileall`, `pip check` и audit зависимостей зелёные.
3. Побайтная parity проверка dashboard RU/EN сохранена в release evidence.
4. GitHub CI зелёный; merge не обходит branch protection.
5. Deploy выполняется только документированным консервативным скриптом без
   `rsync --delete` по умолчанию.
6. После deploy проходят `/ready`, API/NDJSON/discovery smoke, content audit,
   RU/EN NLP audit и визуальная проверка ключевых страниц.
7. После первого реального цикла worker пустой, но доступный источник получает
   `freshness_basis=source_check`; ошибки остаются только в operator surface.
